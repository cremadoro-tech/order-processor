## ワークフロー・オーケストレーション

### 1. プランモードをデフォルトに

- 3ステップ以上、または設計判断が必要なタスクは必ずプランモードで開始する
- 問題が発生したらすぐに止まって再計画する（そのまま突き進まない）
- 構築だけでなく検証ステップにもプランモードを使う
- 曖昧さを減らすために、最初に詳細な仕様を書く

### 2. サブエージェント戦略

- メインのコンテキストウィンドウをきれいに保つためサブエージェントを積極的に活用する
- リサーチ・調査・並列分析はサブエージェントに任せる
- 複雑な問題にはサブエージェントで計算リソースを追加投入する
- 1つのサブエージェントは1つのタスクに集中させる

### 3. 自己改善ループ

- ユーザーから修正を受けたら必ず `tasks/lessons.md` にパターンを記録する
- 同じミスを繰り返さないためのルールを自分で書く
- ミス率が下がるまで教訓を徹底的に反復する
- セッション開始時に関連プロジェクトの教訓を見直す

### 4. 完了前の検証

- 動作を証明せずにタスク完了にしない
- 「シニアエンジニアがこれを承認するか？」と自問する
- テスト実行・ログ確認・正確さの実証を行う
- 変更前後の差分を確認する

### 5. エレガントさを追求（バランスよく）

- 非自明な変更には「もっとエレガントな方法はないか？」と立ち止まる
- ハックっぽい修正なら「今の知識で最もエレガントな解決策を実装する」
- 単純・明白な修正には過剰設計しない
- 自分の作業を提示する前に自己レビューする

### 6. 自律的なバグ修正

- バグ報告を受けたら自分で直す（手取り足取り教えてもらわない）
- ログ・エラー・失敗テストを確認して解決する
- ユーザーのコンテキストスイッチを最小化する
- CIテストの失敗は言われなくても自分で修正する

## タスク管理

1. **まず計画**: チェック可能な項目で `tasks/todo.md` に計画を書く
2. **計画の確認**: 実装開始前に計画をチェックインする
3. **進捗の記録**: 完了した項目をリアルタイムでマークする
4. **変更の説明**: 各ステップで高レベルのサマリーを提供する
5. **結果のドキュメント化**: `tasks/todo.md` にレビューセクションを追加する
6. **教訓のキャプチャ**: 修正後に `tasks/lessons.md` を更新する

## メモリ運用ルール（`00_Important_Context/memories/`）

このディレクトリには、セッションをまたいで参照すべき重要情報が格納されている。**作業開始時や判断が必要な場面で必ず参照すること。**

| ファイル | 用途 | 参照タイミング |
|:--------|:-----|:-------------|
| `preferences.md` | 文体の好み、仕事の進め方、ツール選好 | コンテンツ作成時・指示の解釈時 |
| `decisions.md` | 大きな意思決定の記録（背景・決定・根拠） | 類似の判断が必要な場面・方針確認時 |
| `context-log.md` | プロジェクト進捗・技術検証結果 | 作業開始時の現状把握・スキル実装状況の確認 |
| `case-judgment-framework.md` | ケース判断の枠組み・判断基準 | 新しい判断パターンが発生した場面 |

### 運用ルール
- **読む**: 作業開始時に `context-log.md` と `preferences.md` を確認する
- **書く**: 大きな意思決定があった場合は `decisions.md` に「背景・決定・根拠」の3点セットで追記する
- **更新**: プロジェクトの進捗や技術検証が完了したら `context-log.md` を更新する

## スキル設計ガイドライン（Anthropic公式準拠）

新しいスキルを作成・修正する際は以下に従う。

### フォルダ構成ルール

```
skill-name/           # kebab-case 必須（スペース・アンダースコア・大文字禁止）
├── SKILL.md           # 必須（大文字小文字厳密: SKILL.md のみ）
├── scripts/           # 任意 - 実行可能コード
├── references/        # 任意 - ドキュメント（トラブルシューティング等）
└── assets/            # 任意 - テンプレート・画像・フォント等
```

**禁止事項:**
- スキルフォルダ内に `README.md` を配置しない（ドキュメントはSKILL.mdまたはreferences/に）
- YAMLフロントマターにXMLタグ（`<` `>`）を含めない
- スキル名に "claude" や "anthropic" を使わない

### YAML フロントマター

```yaml
---
name: skill-name          # kebab-case, フォルダ名と一致
description: |            # WHAT + WHEN + Capabilities の3要素を含める
  [何をするか] + [いつ使うか（トリガーフレーズ）] + [主な機能]
---
```

**descriptionの書き方:**
- 「何をするか」を先に、次に「Use when user says "..."」でトリガーフレーズを列挙
- ユーザーが実際に言いそうなフレーズを日本語・英語両方で含める
- 1024文字以内

### Progressive Disclosure（3段階開示）

1. **第1段階（YAMLフロントマター）**: 常にシステムプロンプトに読み込まれる。スキルの使用判断に必要な最小限の情報のみ
2. **第2段階（SKILL.md本文）**: スキルが関連すると判断された時に読み込まれる。完全な指示と手順
3. **第3段階（references/内ファイル）**: 必要に応じてClaude が参照。API詳細・トラブルシューティング等

→ SKILL.mdは5,000語以内を目安に。詳細ドキュメントは `references/` に分離してリンクする

### スキル品質チェックリスト

- [ ] フォルダ名がkebab-case
- [ ] SKILL.mdが正確な命名
- [ ] descriptionにWHAT + WHENが含まれている
- [ ] README.mdがスキルフォルダ内にない
- [ ] 指示が具体的かつ実行可能（「適切に検証する」ではなく具体的な手順を記載）
- [ ] エラーハンドリングが含まれている
- [ ] 使用例が提供されている

## EC・ストア運営固有のルール

- **本番影響に注意**: 変更がライブストアの商品リストに影響する可能性がある。デプロイ前に必ず確認する
- **マルチプラットフォーム考慮**: 共有コードを変更する際は楽天・Yahoo・Amazon・Shopify等への影響を考慮する
- **データ整合性優先**: 商品コード・SKU・バリエーション管理は慎重に。1件のミスが複数プラットフォームに波及する
- **GASスクリプト**: Google Apps Scriptの変更は必ずステージング相当で確認してから本番反映する

## 注意事項・コア原則

**命名規則・コード規約・禁止事項の詳細は `RULES.md` を参照。** 以下は最重要ルール：

- **機密情報の扱い**: APIキー・認証情報はファイルに直接書き込まない。`.env` ファイルを使用する
- **顧客情報**: 売上データ・顧客情報は外部共有しない
- **Git管理外**: `.env`、`*.key`、`credentials/` フォルダはGit管理対象外
- **推測で進めない**: 不明な点があれば必ずユーザーに確認してから作業する
- **シンプルさ優先**: 変更は最小限に。影響範囲を小さく保つ
- **手抜きなし**: 根本原因を探る。一時しのぎ禁止。シニア開発者基準で取り組む
- **最小インパクト**: 必要な箇所だけ触る。新たなバグを持ち込まない
- **透明性**: 何をしているか、なぜそうするかを常に日本語で説明する
